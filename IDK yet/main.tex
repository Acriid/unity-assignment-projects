\section{General description}
The prototype will be a 2D top down game in which the player will be given the task
of escaping the \gls{map} that they were put in.

\section{Mechanics}
The player is able to move and interact with the environment, the main environment
interactions will be picking up an object for 

\subsection{Actions and Challenges}
The player will be able to move around in a 2D environment and interact with objects
within that environment. 
The player may face the challenge of if they should listen to the dialog system or not
along with moving objects around in order to solve puzzles 




\subsection{Genre}
The genre of the game would be a Narrative game.
Some sub-genres include: Puzzle, Horror.


\subsection{Game examples}
\subsubsection*{The Stanley Parable: Different endings and narrator}
In The Stanley Parable the player is guided through the game by a narrator who tells
the player what the player should do. The player can decide to listen to the narrator
or go against the narrator. Different endings can be achieved based off of if the 
player listened to the narrator or not.
The dialog system in the prototype will work similar to the narrator in Stanley Parable
where the system will guide the player to a specific outcome and the player can decide
if they want to listen or not.
\subsubsection*{DarkWood: Fog of war}
The Fog of war in DarkWood only allows the player to see the general shape of the \gls{map}
while showing the player only a small area around them along with where the character is 
looking. The Fog of war system will work similar to the one in DarkWood but the player
will only be allowed to see what is around their character.



\section{Interrogation}

\subsection{Hypothesis}
The player will explore other aspects of the game that they have not found if
they know there is more to find even when a clear winning goal is shown.

\subsection{Goals}
To spark the player's curiosity through a dialog system that will speak to the
player.
To limit the player's vision through either a fog of war system or a field of view system
in order to get the player curios about what they cannot see.


\section{Process}
User input code has been used from a previous project.
Item pooling code was used from a previous project for the dialog.
A generic tilemap was made in asseprite to use for a \gls{map}
\newline
Different ways fog of war was tried:
\newline
A RenderTexture was used along with shaders 
that paint the texture white. 2 Different shaders were used the FogOfWar
and vision. The FogOfWar shader is a completely black shader that then checks for
if there is any vision material and will then change the RenderTexture to white in
the area that the material is allowing the player to see that area. The reason that
this solution did not work was the uv's between the FogOfWar camera and the Main camera
where not aligning, which caused the vision circle to be stretched.
Thus a single area light was used for the player to see around them to simulate a
FogOfWar. This solution was chosen because of time constraints, thus it is more
CPU based instead of the RenderTexture which is more GPU based.

\section{Player motivation design}
The player's actions may also change depending on what the dialog system tells them
for example if the dialog system tells the player to go left the player may just be
motivated to go right.


Make the player curious to keep them motivated. The player will be motivated by the
dialog system telling them that there is still something more to do even after the 
player achieved an ending. If the dialog system is fleshed out and does dialog based
on what the player does in the game then the player will try new actions in order to
see what the dialog system will tell them.
